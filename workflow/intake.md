# intake.md — 入库流程（每次研究收尾必走）

> 触发时机：任何一次研究 / 分析 / 逆向 / 实验收尾，**且产生了值得留存的结论**。
> 目标：把结论可靠地沉淀成一个知识页，别让它只留在对话里。

## 总闸（开始前先读）

- **必须为下面第 0 到第 6 步各建一个 todo，按顺序完成，禁止跳步。**
- **红线：只要有一步没过，这个页面的 `status` 不许标 `active`。**
- 所有规则细节以 `schema.md` 为准（frontmatter、kind、status、tag、命名、可见性）。
- 违反某一步的字面要求，就是违反这套流程——别用「精神上做到了」绕过。

## 第 0 步 · 前置门：值不值得入库

- 这次研究有没有「下次能直接读」的明确结论？
  - 有 → 继续。
  - 没有 → 停。别为凑数入库。
- 反合理化：「这条太小不值得记」→ 小结论正是知识漂移的起点，该记就记；确实没结论才停。

**没过这步 → 停。**

## 第 1 步 · 选类型

- 从 `concept` / `mechanism` / `case` / `decision` / `investigation` 选一个（定义见 `schema.md` 第 3 节）。
- 打开 `workflow/templates/` 里对应的模板。

**没选类型（= 没有模板可填）→ 停。**

## 第 2 步 · 填 frontmatter + 写正文

- 知识页统一放公开根 `analysis/`（见 `schema.md` 第 1 节）。`private/` 只是本地非公开材料区（脚本、一次性草稿），不是知识库根、不走本流程；确有不能公开的结论时先停下来跟维护者确认怎么归置，别默认塞进 `private/` 当知识页。
- 套模板，六个 frontmatter 字段一个不能缺：`title` / `kind` / `status` / `updated` / `applies_to` / `tags`。
- 先填 `status: draft`（成不成熟第 6 步再判）。
- 禁止占位符：`tags` 写 `TODO`、`applies_to` 留空，都算没填。

**缺任一必填字段 → 停。**

## 第 3 步 · 冲突复检（硬门，不可跳过、不可降级）

- **必须派一个全新上下文的子代理**去查冲突，不能让写这页的自己查自己。
- 复检对象：同类旧笔记、相同 tags、相同主题、可能被本结论推翻的 `active` 页面；同时做下方的过期建议（`STALE_SUGGESTION`）。
- 为什么必须换个干净子代理：写这页的 AI 自查，通常不会承认自己的疏漏（三家框架的共识）。

### 3.1 冲突裁决（硬门）

三级裁决，**只回答「新旧结论有没有矛盾」**：

- **BLOCKER**（推翻了某个 `active` 旧结论）→ 停。先把旧页标 `superseded` 或 `stale`，再回来。
- **WARNING**（部分矛盾 / 存疑）→ 显式浮出来给维护者拍板，绝不自动放行。
- **OK**（无冲突）→ 可继续后续步骤（仍须完成 3.2 的输出义务）。

无论裁决是哪一级，**都必须列出本轮实际检查过的候选页路径**（可审计；禁止空转后只写 `OK`）：

- 有候选 → `检查过: path/a.md, path/b.md`（只写路径，不要求摘录正文）。
- 确实没有合适候选 → `检查过: 无合适候选（理由：…）`，禁止省略。
- 候选范围与 3.2 防噪一致：优先同 `topic:`、同主题路径、或 `applies_to` 同产品/工具族的 `active` 页。

### 3.2 过期建议 `STALE_SUGGESTION`（软门，必输出）

冲突 ≠ 旧页仍有效。子代理在冲突裁决之外，**必须**再判断：

> 本页结论是否让某个已有 `active` 页的事实基础过时（版本号、工具/API 行为、外部事实等），即便新旧结论并不直接互斥。

规则：

- **范围（防噪）**：优先同 `topic:` 标签、同主题路径、或 `applies_to` 同产品/工具族的 `active` 页。不要把「任意一页有版本号」都标过期。
- **输出**：无则写 `STALE_SUGGESTION: 无`；有则每条写清路径 + 一句话理由（建议标 `stale` 或复核，不自动改状态）。
- **卡点**：**不因** `STALE_SUGGESTION` 单独停入库。但必须把清单浮给维护者；处理旧页（标 `stale` / 改 `applies_to` / 显式「暂不处理」）是维护者职责，写页的 agent 不得静默吞掉。
- **与 BLOCKER 的区别**：BLOCKER = 结论打架，必须先处理；`STALE_SUGGESTION` = 旧页可能该过期，默认只建议。

子代理结果建议固定成两段（便于扫日志）：

```text
冲突裁决: OK | WARNING | BLOCKER
  检查过: path/a.md, path/b.md
  （若 WARNING/BLOCKER：涉及的旧页路径 + 理由）
  （若无候选：检查过: 无合适候选（理由：…））

STALE_SUGGESTION: 无
  （或）
STALE_SUGGESTION:
  - path/to/old.md — 理由：…
```

**没做复检、或有 BLOCKER 没处理 → 停。**  
**做了复检但未输出 `检查过` 行 → 停（复检不可审计）。**  
**做了复检但未输出 `STALE_SUGGESTION` 段（含「无」）→ 停（复检不完整）。**

## 第 4 步 · 更新索引

- 往 `analysis/index.md` 加一行：相对路径 + 一句话说明 + 标签。
- 硬话（消费机制）：`index` 不更新 = 这页检索不到 = 等于没入库。

**没更新 `index.md` → 停。**

## 第 5 步 · 写日志

- 往 `analysis/log.md` 追加一行，固定格式：

  ```text
  [2026-07-06] kind | 一句话结论 | 推翻旧结论？（无 / superseded: 旧页名）
  ```

- 只追加，不回改历史。没记就丢。

**没写 `log.md` → 停。**

## 第 6 步 · 标签校验 + status 判定

- 对照 `glossary.md`：标签是否都在术语表内？有没有同义词漂移（同一个东西两种叫法）？新词先加进 `glossary.md` 再用。
- **status 判定（六步跑完不等于 active）**：
  - 依据充分 + 边界清楚 + 可复核 + 第 3 步复检通过 → 标 `active`。
  - 还有未确认点 / 证据不足 / 待人工判断 → 保留 `draft`。
  - 依据已过时 → 标 `stale`。
- 只有判定为 `active`，才算真正入库完成。

**标签有漂移未处理，或没做 status 判定 → 停。**
